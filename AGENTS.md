# AGENTS.md

Этот документ описывает правила работы агента, который генерирует недельный план питания (weekly meal plan) на основе рецептов в этом репозитории.

Документ должен использоваться как основной источник правил для агента.

---

## Профиль пользователя

Этот блок описывает человека, для которого составляется план питания.
При необходимости этот блок можно заменить для другого пользователя.

### Основные данные

- пол: мужчина
- возраст: 34
- рост: 175 см
- вес: 86 кг

### Цель

Постепенное снижение веса.
Целевой диапазон веса: 81–82 кг.
Стратегия: умеренный дефицит калорий без жёстких ограничений.

### Целевые калории

Ориентир суточной калорийности: 2100–2300 ккал
План не должен системно опускаться значительно ниже этого диапазона.

### Целевые КБЖУ на день

Базовый ориентир на день:

- калории: около 2200 ккал
- белки: около 170 г
- жиры: 65–70 г
- углеводы: 220–230 г

Это ориентиры, а не требование к математически точному совпадению каждый день. Однако при выборе рецептов и сборке недели агент должен стремиться держать план вблизи этих значений и избегать системного перекоса в сторону недобора белка или избытка жиров.

### Физическая активность

- муай-тай: 2 тренировки в неделю
- силовые тренировки: 1 в неделю
- кардио тренировки: 2-3 часа в неделю на беговой дорожке

### Образ жизни

- пользователь живёт один
- высокая рабочая нагрузка
- желательно минимизировать время готовки
- питание должно быть простым и повторяемым

### Ограничения

- частичная непереносимость лактозы
- склонность к изжоге / рефлюксу

### Явные запреты

Не использовать:

- спортивный протеин
- протеиновые порошки
- протеиновые шейки

Белок должен поступать из обычных продуктов.

---

## Цель системы

Агент должен создавать реалистичный недельный план питания, который:

- можно соблюдать длительное время
- не вызывает постоянного голода
- не требует сложной готовки
- минимизирует пищевые отходы
- подходит для одного человека

Система оптимизирована для устойчивого питания, а не для агрессивной диеты.

---

## Структура репозитория

Рецепты находятся в директории:

```
recipes/
  breakfasts/
  lunches/
  dinners/
  snacks/
```

Каждый рецепт расположен в отдельной директории:

```
recipes/<meal_type>/<recipe_id>/
  recipe.yml
  README.md
```

`recipe.yml` — источник структурированных данных.  
`README.md` — описание рецепта для человека.

`README.md` должен оставаться синхронизированным с `recipe.yml`:

- название рецепта в `README.md` должно совпадать с `name` из `recipe.yml`
- ингредиенты и их количества должны соответствовать `recipe.yml`
- КБЖУ в `README.md` должны соответствовать `nutrition_per_serving`

---

## Структура рецепта

Файл `recipe.yml` должен содержать поля:

```
id
name
meal_type
servings
prep_time_minutes
ingredients
nutrition_per_serving
```

### Типы приёмов пищи

Допустимые значения `meal_type`:

```
breakfast
lunch
dinner
snack
```

### servings

servings — реальное количество порций, которое получается из рецепта.

Количество ингредиентов должно соответствовать этому числу.

---

## Структура ингредиентов

Каждый ингредиент должен иметь поля:

```
key
name
amount
unit
pantry (опционально)
```

Пример:

```
- key: chicken_breast
  name: Куриное филе
  amount: 300
  unit: g
```

### key

key используется для агрегации ингредиентов при формировании списка покупок.

Требования:

- snake_case
- стабильный идентификатор

---

## Pantry-ингредиенты

Некоторые ингредиенты считаются базовыми и обычно есть дома.

Примеры:

- масло
- соль
- перец
- специи
- соевый соус

Если ингредиент имеет поле `pantry: true` он остаётся в рецепте, но не попадает в список покупок.

---

## Реалистичные размеры порций

Использовать следующие ориентиры:

курица / индейка: 150–200 г на порцию  
рыба: 150–200 г на порцию  
рис / гречка (сухие): 70–80 г на порцию  
яйца: 2–3 шт на порцию  
перекусы: примерно 150–300 ккал

Эти значения используются как sanity-check при создании рецептов.

---

## Структура дня

Каждый день должен содержать:

- breakfast
- snack_1
- lunch
- snack_2
- dinner

Перекусы обязательны — они уменьшают риск сильного голода и переедания вечером.

## Стандартизация слотов по КБЖУ

Чтобы рецепты внутри одного типа приёма пищи можно было свободно менять местами, агент должен стремиться держать их в узких коридорах по КБЖУ.

Это не значит, что каждый рецепт обязан иметь одинаковые числа до грамма, но внутри одного типа рецепты должны быть близки настолько, чтобы замена `breakfast -> breakfast`, `lunch -> lunch`, `dinner -> dinner`, `snack -> snack` не ломала день по макросам.

Целевые коридоры на 1 порцию:

- breakfast: 450–550 ккал, белки 30–35 г, жиры 10–16 г, углеводы 45–60 г
- lunch: 600–700 ккал, белки 42–52 г, жиры 15–22 г, углеводы 55–75 г
- dinner: 500–620 ккал, белки 38–50 г, жиры 14–24 г, углеводы 30–60 г
- snack: 200–300 ккал, белки 12–28 г, жиры 5–12 г, углеводы 18–30 г

Если рецепт заметно выходит за пределы коридора, агент должен по возможности:

- скорректировать состав и размеры порций
- уменьшить жирные ингредиенты и добавить более предсказуемый гарнир или белок
- выровнять рецепт относительно остальных рецептов этого же meal type

---

## Гибкость перекусов

Перекусы являются гибкой частью плана.

Если после приготовления рецептов остаются продукты (например йогурт, кефир, ягоды, хлеб, авокадо), допустимо заменить запланированный перекус на использование этих продуктов.

Это помогает уменьшить пищевые отходы и не нарушает структуру плана.

---

## Овощи по умолчанию

Считать по умолчанию:

обед содержит примерно 200–300 г овощей  
ужин содержит примерно 150–250 г овощей

Отдельный слот для овощей в plan.yml создавать не нужно.

---

## Логика разнообразия внутри недели

Нужно избегать двух крайностей:

- слишком много разных блюд
- одно и то же блюдо каждый день

План должен быть повторяемым, но не монотонным.

Базовые ориентиры на неделю:

- завтраки: 3–4 разных рецепта
- обеды: 2–3 разных рецепта
- ужины: 2–3 разных рецепта
- перекусы: 5–7 разных рецептов

Допускается повторение одного и того же блюда 2–3 раза в течение недели,
если это часть нормальной batch-логики и уменьшает количество готовки.

Для быстрых приёмов пищи, которые не готовятся сразу на несколько порций, правила жёстче:

- один и тот же завтрак, который не готовится сразу на несколько порций, обычно не должен появляться больше 2 раз в неделю
- один и тот же перекус обычно не должен появляться больше 2 раз в неделю
- 3 появления допустимы только если это оправдано логикой остатков, уже открытого скоропортящегося продукта или готовых порционных единиц, которые уже есть в наличии

Внутри одного дня агент должен избегать дублирования одной и той же продуктовой базы.

Нежелательные примеры:

- творог на завтрак и творог в одном из перекусов в тот же день
- лосось на завтрак и лосось в перекус в тот же день
- йогуртовый завтрак и йогуртовый перекус в тот же день

Если в одном слоте дня уже есть выраженная база или основной белок,
в остальных слотах этого дня по возможности нужно выбирать другие базы.

Особенно важно чередовать:

- творог / йогурт / мягкие молочные продукты
- лосось / рыбу
- яйца
- индейку / курицу

Хороший ориентир на неделю — распределять её хотя бы между 3 опорными белками, например:

- курица
- индейка
- рыба / лосось

---

## Минимизация готовки

Агент должен:

- использовать рецепты на несколько порций
- распределять эти порции по дням
- уменьшать количество cooking-сессий

Цель — минимальная готовка в рабочую неделю.

---

## Использование порций

Если рецепт имеет `servings > 1`, приготовленные порции должны быть
распределены по нескольким дням недели.

Пример:

Рецепт: servings = 3

Допустимое распределение:

понедельник — обед  
вторник — обед  
среда — обед

Не следует готовить блюдо на несколько порций,
если часть порций останется неиспользованной.

К концу недели не должно оставаться приготовленных блюд.

---

## Логика ингредиентов, упаковок и отходов

Пользователь живёт один, поэтому агент должен минимизировать отходы через разумное переиспользование ингредиентов, а не через монотонное меню.

При выборе недели агент должен сначала сформировать ограниченный набор core ingredients, вокруг которых строится план.

Обычно это:

- 3–4 опорных белка
- 3–4 основных углеводных гарнира
- ограниченный набор повторяющихся овощей

Если несколько рецептов подходят одинаково хорошо, нужно предпочитать те, которые используют уже выбранные ингредиенты недели.

Если используется продукт вроде:

- лосося
- творожного сыра
- авокадо
- ягод
- хлеба

желательно использовать его повторно в течение 1–3 дней.

### Открытые и скоропортящиеся продукты

При распределении рецептов по неделе агент должен учитывать не только суммарное количество ингредиента, но и реалистичный сценарий покупки, вскрытия упаковки и срока жизни после этого.

Особенно важно для:

- слабосолёного лосося
- свежей рыбы
- авокадо
- листьев салата
- ягод
- мягких молочных продуктов после вскрытия

Если продукт обычно покупается одной упаковкой и после вскрытия его качество заметно падает через 1–3 дня, его использования нужно ставить плотным блоком, а не размазывать по всей неделе.

Пример:

- если слабосолёный лосось обычно покупается упаковкой около 300 г, не нужно ставить тосты с лососем во вторник, четверг, субботу и воскресенье
- такие приёмы пищи должны идти подряд или почти подряд в пределах 1–3 дней после покупки / вскрытия
- если продукт нужен и в начале, и в конце недели, это должно означать две отдельные покупки, а не одну условно "вечную" упаковку

Агент не должен молча предполагать, что открытый скоропортящийся продукт нормально доживёт с понедельника до воскресенья.

### Порционные продукты

Некоторые продукты обычно продаются не одной большой вскрытой ёмкостью,
а отдельными порциями.

Примеры:

- йогурт в баночках
- готовые порционные кефиры
- индивидуальные творожные или десертные стаканчики

Для таких продуктов агент должен считать, что одна порция рецепта обычно
закрывается одной отдельной упаковкой.

Следствия:

- не нужно ставить один и тот же йогуртовый перекус много дней подряд только ради того, чтобы "добить" продукт
- порционный йогурт не считается проблемным остатком, который обязательно надо растянуть на всю неделю
- повторение йогурта допустимо, но только как нормальный выбор перекуса, а не как побочный эффект упаковки
- если в неделе уже есть 1–2 йогуртовых перекуса, остальные перекусы стоит по возможности добирать другими вариантами, если для этого не требуется лишняя сложная готовка

Иначе говоря:

- большая вскрытая упаковка или открытый скоропортящийся продукт -> лучше сгруппировать использования
- отдельные порционные единицы -> не нужно искусственно повышать частоту появления в меню

---

## Разнообразие между неделями

При генерации нового недельного плана агент должен просмотреть несколько предыдущих недель,
если они существуют.

Предпочтительный горизонт:

- сначала предыдущая неделя
- при наличии ещё 1–3 более ранних недель тоже учитывать их

Точная копия прошлой недели по набору рецептов недопустима.

Приоритеты ротации:

- обеды и ужины нужно ротировать строже, чем быстрые завтраки и перекусы
- если рецепт горячего блюда был в прошлой неделе, в новой неделе лучше выбрать другой рецепт того же meal type
- если рецепт уже использовался 2 недели подряд, третью неделю подряд его лучше не ставить, кроме случаев, когда выбор в репозитории пока ограничен

Повтор между неделями допустим для:

- простых базовых завтраков
- быстрых перекусов
- очень удобных блюд, которые хорошо работают по КБЖУ и закупкам

Но даже в этих случаях новая неделя не должна ощущаться копией предыдущей.

Хороший минимум:

- заменить хотя бы 1 завтрак по сравнению с прошлой неделей
- заменить хотя бы 1 горячее блюдо по сравнению с прошлой неделей
- обновить хотя бы часть пула перекусов, если это не увеличивает заметно сложность закупок

---

## Алгоритм выбора рецептов

При составлении плана агент должен:

1. Прочитать все рецепты из `recipes/`.
2. Прочитать предыдущие meal plan, как минимум за прошлую неделю, а при наличии ещё 1–3 более ранних недель — тоже учесть их для ротации.
3. Разделить рецепты по типу приёма пищи.
4. Выбрать core ingredients недели: основные белки, гарниры и повторяющиеся овощи.
5. Выбрать ограниченный пул рецептов на неделю:
   - 3–4 завтрака
   - 2–3 обеда
   - 2–3 ужина
   - 5–7 перекусов
6. Предпочитать рецепты с общими ингредиентами.
7. Предпочитать рецепты с небольшим временем приготовления.
8. Распределить batch-блюда по нескольким дням подряд с учётом количества порций.
9. Проверить логику упаковок, сроков хранения и открытых продуктов до финального распределения по дням.
10. Проверить, что внутри одного дня не дублируется одна и та же база или основной белок без необходимости.
11. Проверить, что новая неделя не является почти точной копией предыдущей.
12. Проверить итоговые КБЖУ по дням и по среднему за неделю, затем при необходимости скорректировать выбор рецептов или распределение.

---

## Проверка качества недели

Перед финализацией weekly plan агент должен проверить итоговую неделю целиком.

### 1. Проверка структуры и повторов

- каждый день должен содержать `breakfast`, `snack_1`, `lunch`, `snack_2`, `dinner`
- batch-блюда могут повторяться 2–3 дня подряд, если это осознанная логика готовки
- быстрые завтраки и перекусы не должны доминировать в неделе без причины
- внутри одного дня не должно быть явного дублирования одной и той же базы или белка, если этого можно избежать

### 2. Проверка КБЖУ

- большинство дней должно попадать примерно в `2100–2300 ккал`
- допустимо небольшое отклонение в отдельный день, если средний день недели остаётся близко к `2200 ккал`
- белок должен быть стабильно высоким; хороший ориентир — около `170 г` в среднем за неделю
- агент должен избегать системного недобора белка и системного избытка жиров
- приёмы пищи по возможности должны оставаться в коридорах своего meal type

### 3. Проверка разнообразия

- неделя не должна состоять из одного и того же завтрака или одного и того же перекуса почти каждый день
- неделя не должна быть перегружена одним белком сразу в горячих блюдах, завтраках и перекусах
- новая неделя не должна быть почти копией предыдущей недели

### 4. Проверка удобства и отходов

- к концу недели не должно оставаться неиспользованных приготовленных порций
- скоропортящиеся продукты должны использоваться плотными блоками
- порционные продукты не должны искусственно раздувать частоту повторов
- список покупок должен быть компактным и логичным для одного человека

---

## Генерация weekly plan

План создаётся в директории `meal-plans/<YYYY-MM-DD>/`, где `<YYYY-MM-DD>` — понедельник соответствующей недели.

Создаются два файла:

```
plan.yml
README.md
```

### plan.yml

`plan.yml` — технический файл.

`plan.yml` является текущим источником истины по неделе.
Если по ходу недели блюда переносятся между днями или заменяются, агент должен обновлять `plan.yml` так, чтобы он отражал фактическое состояние недели, а не только исходный черновой план.

Если по ходу работы меняется рецепт и его КБЖУ, агент должен пересчитывать только текущую неделю и будущие недели, где этот рецепт ещё актуален, включая текущий день изменения.

Прошлые недели, которые уже являются историей, автоматически переписывать не нужно. Их следует менять только по прямому запросу пользователя.

Для каждого дня указываются:

```
breakfast
snack_1
lunch
snack_2
dinner
```

Обычный приём пищи записывается как рецепт:

```yml
lunch:
  recipe_id: pasta_boloneze
  servings: 1
```

Если приём пищи был пропущен, использовать:

```yml
snack_2:
  status: skipped
```

Если вместо планового приёма пищи была еда не из репозитория,
использовать технический статус `cheat_meal`:

```yml
lunch:
  status: cheat_meal
  estimated_calories: 600
  note: Голубцы у мамы
```

`cheat_meal` здесь означает не обязательно "срыв" или "вредную еду", а любой приём пищи вне репозитория, который не хочется заводить отдельным рецептом.

Это может быть:

- домашняя еда у родственников
- еда в гостях
- заказ или ресторан
- любое разовое блюдо вне репозитория

Для `cheat_meal`:

- `estimated_calories` обязателен как грубая реалистичная оценка калорийности
- `note` желателен, если известно что именно было съедено; это короткое описание блюда или контекста, например `Голубцы у мамы` или `Утка по-пекински и токпокки со свининой и ананасами`
- не нужно подробно расписывать состав и БЖУ, если для этого нет удобной и достаточно реалистичной оценки
- если одно и то же внешнее блюдо начинает повторяться регулярно и его удобно учитывать, лучше оформить отдельный рецепт в репозитории

В `plan.yml` не должно быть пояснений, комментариев или лишних текстовых заметок.

### README.md

`README.md` — человекочитаемый план недели.

Он должен содержать:

- 7 дней меню
- блюда и порции
- id рецептов в формате `type/id` (например, `breakfast/tosty_s_lososem`)
- Markdown-ссылки на `README.md` рецептов в меню недели и в плане готовок для быстрого перехода
- КБЖУ каждого приёма пищи из репозитория
- суммарные КБЖУ дня, если они доступны
- план готовок
- список закупок
- агрегированный список продуктов

Если в `plan.yml` слот помечен как `status: skipped`, этот приём пищи можно не показывать в `README.md`.

Если слот помечен как `status: cheat_meal`, в `README.md` нужно показывать нейтральную краткую запись про внешнюю еду, а не обязательно слово "читмил".

Предпочтительный формат:

- если есть `note`: `Еда вне репозитория — голубцы у мамы, примерно 600 ккал`
- если `note` нет: `Еда вне репозитория — примерно 1300 ккал`

Если для такой внешней еды нет адекватной оценки БЖУ, допустимо показывать только калории для этого приёма пищи и только калории в сумме за день.

## Правила составления плана закупок

План закупок должен строиться в два этапа:

1. Сначала агрегировать все ингредиенты на всю неделю.
2. Потом делить их на закупки по сроку хранения и разумности, а не просто по дням использования.

План закупок является ориентиром на момент генерации недели, а не строгим источником истины после всех последующих изменений.

Если по ходу недели `plan.yml` корректируется под фактическое питание, агент не обязан автоматически пересчитывать закупки или приводить их в полное соответствие с обновлённым фактом, если пользователь отдельно этого не просил.

### Категории закупки

#### 1. Купить сразу на неделю

Продукты, которые нормально хранятся 5–7 дней и дольше.

Примеры:

- яйца
- лук
- морковь
- картофель
- рис
- гречка
- паста
- овсянка
- гранола
- орехи
- арахисовая паста
- томатный соус

Такие продукты не нужно дробить на несколько закупок.

#### 2. Купить в первой половине недели и при необходимости докупить

Продукты со средним сроком годности.

Примеры:

- хлеб
- йогурт
- кефир
- молочные продукты
- помидоры
- огурцы
- кабачки
- брокколи
- ягоды
- зелень

#### 3. Купить ближе к дню приготовления

Очень скоропортящиеся продукты или продукты, которые лучше брать максимально свежими.

Примеры:

- свежее филе рыбы
- листья салата
- некоторые готовые молочные продукты с коротким сроком хранения

### Дополнительные правила

- Не дробить стабильные ингредиенты на несколько закупок.
- Яйца, крупы, пасту, лук, морковь и подобные продукты покупать сразу суммарно на неделю.
- Если ингредиент обычно продаётся упаковкой, допустимо округлять до упаковки.
- План закупок должен быть удобен для человека, а не только математически точен.
- Неделя должна по возможности укладываться в 1 основную закупку и 0–2 небольшие докупки скоропортящихся продуктов, а не требовать почти ежедневных походов в магазин.
- Если ингредиент скоропортящийся и обычно покупается одной упаковкой, размер упаковки должен влиять не только на список покупок, но и на распределение блюд по дням.
- Если продукт обычно продаётся отдельными порциями, не нужно искусственно группировать или учащать его в плане только ради "использования упаковки".

Если продукт обычно продаётся упаковкой, допустимо округлять количество до ближайшей упаковки.

Примеры:

- яйца: 15 → купить упаковку 20
- порционный йогурт: 3 перекуса по 150 г → купить 3 баночки, а не пытаться растянуть один йогурт на всю неделю
- кефир: 250 мл → купить бутылку
- авокадо: 140 г → купить 1–2 штуки
- слабосолёный лосось: около 280–320 г → купить 1 упаковку примерно 300 г и использовать её в пределах 1–3 дней, либо планировать отдельную более позднюю покупку

## Добавление новых рецептов

Чтобы добавить рецепт:

1. создать директорию `recipes/<meal_type>/<recipe_id>/`
2. создать файлы `recipe.yml` и `README.md`
3. соблюдать структуру `recipe.yml`
4. использовать реалистичные порции
5. избегать сложных рецептов
6. не использовать протеиновые порошки

---

## Что агент делать не должен

Агент не должен:

- придумывать рецепты вне репозитория
- генерировать экстремально низкокалорийные планы
- строить меню из одного блюда на всю неделю
- вводить сложные системы учёта калорий или ингредиентов

---

## Критерий хорошего результата

Хороший meal plan — это план, который пользователь:

- реально соблюдает
- может готовить при рабочей нагрузке
- может легко закупить
- не бросает к середине недели
