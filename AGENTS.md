# AGENTS.md

Этот документ описывает правила работы агента, который генерирует недельный план питания (weekly meal plan) на основе рецептов в этом репозитории.

Документ должен использоваться как основной источник правил для агента.

---

## Профиль пользователя

Этот блок описывает человека, для которого составляется план питания.
При необходимости этот блок можно заменить для другого пользователя.

### Основные данные

- пол: мужчина
- возраст: 34
- рост: 175 см
- вес: 86 кг

### Цель

Постепенное снижение веса.
Целевой диапазон веса: 81–82 кг.
Стратегия: умеренный дефицит калорий без жёстких ограничений.

### Целевые калории

Ориентир суточной калорийности: 2100–2300 ккал
План не должен системно опускаться значительно ниже этого диапазона.

### Физическая активность

- муай-тай: 2 тренировки в неделю
- силовые тренировки: 1 в неделю
- кардио тренировки: 2-3 часа в неделю на беговой дорожке

### Образ жизни

- пользователь живёт один
- высокая рабочая нагрузка
- желательно минимизировать время готовки
- питание должно быть простым и повторяемым

### Ограничения

- частичная непереносимость лактозы
- склонность к изжоге / рефлюксу

### Явные запреты

Не использовать:

- спортивный протеин
- протеиновые порошки
- протеиновые шейки

Белок должен поступать из обычных продуктов.

---

## Цель системы

Агент должен создавать реалистичный недельный план питания, который:

- можно соблюдать длительное время
- не вызывает постоянного голода
- не требует сложной готовки
- минимизирует пищевые отходы
- подходит для одного человека

Система оптимизирована для устойчивого питания, а не для агрессивной диеты.

---

## Структура репозитория

Рецепты находятся в директории:

```
recipes/
  breakfasts/
  lunches/
  dinners/
  snacks/
  _template/
```

Каждый рецепт расположен в отдельной директории:

```
recipes/<meal_type>/<recipe_id>/
  recipe.yml
  README.md
```

`recipe.yml` — источник структурированных данных.  
`README.md` — описание рецепта для человека.

Директория `_template` используется только как пример структуры.

---

## Структура рецепта

Файл `recipe.yml` должен содержать поля:

```
id
name
meal_type
servings
prep_time_minutes
ingredients
nutrition_per_serving
```

### Типы приёмов пищи

Допустимые значения `meal_type`:

```
breakfast
lunch
dinner
snack
```

### servings

servings — реальное количество порций, которое получается из рецепта.

Количество ингредиентов должно соответствовать этому числу.

---

## Структура ингредиентов

Каждый ингредиент должен иметь поля:

```
key
name
amount
unit
pantry (опционально)
```

Пример:

```
- key: chicken_breast
  name: Куриное филе
  amount: 300
  unit: g
```

### key

key используется для агрегации ингредиентов при формировании списка покупок.

Требования:

- snake_case
- стабильный идентификатор

---

## Pantry-ингредиенты

Некоторые ингредиенты считаются базовыми и обычно есть дома.

Примеры:

- масло
- соль
- перец
- специи
- соевый соус

Если ингредиент имеет поле `pantry: true` он остаётся в рецепте, но не попадает в список покупок.

---

## Реалистичные размеры порций

Использовать следующие ориентиры:

курица / индейка: 150–200 г на порцию  
рыба: 150–200 г на порцию  
рис / гречка (сухие): 70–80 г на порцию  
яйца: 2–3 шт на порцию  
перекусы: примерно 150–300 ккал

Эти значения используются как sanity-check при создании рецептов.

---

## Структура дня

Каждый день должен содержать:

- breakfast
- snack_1
- lunch
- snack_2
- dinner

Перекусы обязательны — они уменьшают риск сильного голода и переедания вечером.

---

## Гибкость перекусов

Перекусы являются гибкой частью плана.

Если после приготовления рецептов остаются продукты (например йогурт, кефир, ягоды, хлеб, авокадо), допустимо заменить запланированный перекус на использование этих продуктов.

Это помогает уменьшить пищевые отходы и не нарушает структуру плана.

---

## Овощи по умолчанию

Считать по умолчанию:

обед содержит примерно 200–300 г овощей  
ужин содержит примерно 150–250 г овощей

Отдельный слот для овощей в plan.yml создавать не нужно.

---

## Разнообразие блюд

Нужно избегать двух крайностей:

- слишком много разных блюд
- одно и то же блюдо каждый день

Допускается повторение одного и того же блюда 2–3 раза в течение недели.
Повторение блюд уменьшает количество готовки и упрощает список покупок.

При этом не следует перегружать неделю одним и тем же основным белком.
Если лосось, курица, индейка или другой основной белок уже используются в горячем блюде 2–3 раза за неделю, не стоит дополнительно ставить этот же продукт ещё в несколько завтраков и перекусов подряд.

Хороший ориентир — распределять неделю хотя бы между 3 опорными белками, например:

- лосось
- индейка
- курица

Ориентиры на неделю:

- завтраки: 3–4
- обеды: 2–3
- ужины: 2–3
- перекусы: 5–7

---

## Минимизация готовки

Агент должен:

- использовать рецепты на несколько порций
- распределять эти порции по дням
- уменьшать количество cooking-сессий

Цель — минимальная готовка в рабочую неделю.

---

## Использование порций

Если рецепт имеет `servings > 1`, приготовленные порции должны быть
распределены по нескольким дням недели.

Пример:

Рецепт: servings = 3

Допустимое распределение:

понедельник — обед  
вторник — обед  
среда — обед

Не следует готовить блюдо на несколько порций,
если часть порций останется неиспользованной.

К концу недели не должно оставаться приготовленных блюд.

---

## Переиспользование ингредиентов

Пользователь живёт один, поэтому важно минимизировать отходы.

Если используется продукт вроде:

- лосося
- йогурта
- творожного сыра
- авокадо

желательно использовать его повторно в течение 1–3 дней.

При выборе рецептов следует предпочитать блюда с общими ингредиентами.

---

## Алгоритм выбора рецептов

При составлении плана агент должен:

1. Прочитать все рецепты из `recipes/`.
2. Разделить их по типу приёма пищи.
3. Выбрать ограниченный пул рецептов на неделю:
   - 3–4 завтрака
   - 2–3 обеда
   - 2–3 ужина
   - 5–7 перекусов
4. Предпочитать рецепты с общими ингредиентами.
5. Предпочитать рецепты с небольшим временем приготовления.
6. Распределить блюда по дням с учётом количества порций.

---

## Генерация weekly plan

План создаётся в директории `meal-plans/<YYYY-MM-DD>/`, где `<YYYY-MM-DD>` — понедельник соответствующей недели.

Создаются два файла:

```
plan.yml
README.md
```

### plan.yml

`plan.yml` — технический файл.

`plan.yml` является текущим источником истины по неделе.
Если по ходу недели блюда переносятся между днями или заменяются, агент должен обновлять `plan.yml` так, чтобы он отражал фактическое состояние недели, а не только исходный черновой план.

Для каждого дня указываются:

```
breakfast
snack_1
lunch
snack_2
dinner
```

Обычный приём пищи записывается как рецепт:

```yml
lunch:
  recipe_id: pasta_boloneze
  servings: 1
```

Если приём пищи был пропущен, использовать:

```yml
snack_2:
  status: skipped
```

Если вместо планового приёма пищи был читмил
или внешняя еда не из репозитория, использовать:

```yml
lunch:
  status: cheat_meal
  estimated_calories: 1300
```

Для `cheat_meal` не нужно подробно расписывать состав блюда, если у пользователя нет удобной и достаточно реалистичной оценки. Достаточно грубой оценки калорийности.

В `plan.yml` не должно быть пояснений, комментариев или лишних текстовых заметок.

### README.md

`README.md` — человекочитаемый план недели.

Он должен содержать:

- 7 дней меню
- блюда и порции
- id рецептов в формате `type/id` (например, `breakfast/tosty_s_lososem`)
- Markdown-ссылки на `README.md` рецептов в меню недели и в плане готовок для быстрого перехода
- КБЖУ каждого приёма пищи из репозитория
- суммарные КБЖУ дня, если они доступны
- план готовок
- список закупок
- агрегированный список продуктов

Если в `plan.yml` слот помечен как `status: skipped`, этот приём пищи можно не показывать в `README.md`.

Если слот помечен как `status: cheat_meal`, в `README.md` нужно показать краткую запись вроде `Читмил — примерно 1300 ккал`.

Если для читмила нет адекватной оценки БЖУ, допустимо показывать только калории для этого приёма пищи и только калории в сумме за день.

### Core ingredients

При составлении недельного плана агент должен
предпочитать рецепты, которые используют повторяющиеся ингредиенты.

Цель — уменьшить разнообразие закупок и снизить пищевые отходы.

Пример core ингредиентов недели:

Белок:

- курица
- индейка
- яйца
- лосось

Углеводы:

- рис
- гречка
- паста
- хлеб

Овощи:

- лук
- морковь
- помидоры
- кабачок
- брокколи

Если несколько рецептов подходят, следует предпочитать те, которые используют уже выбранные ингредиенты недели.

## Разнообразие между неделями

При генерации нового недельного плана агент должен просмотреть предыдущий план питания (если он существует).

Желательно избегать точного повторения тех же рецептов.

Если в предыдущей неделе использовались рецепты:

- lunch/kurica_ris
- lunch/grechka_indeyka
- dinner/losos_ovoshi

следует по возможности выбрать другие рецепты из репозитория.

Однако это мягкое правило.

Допустимо повторять:

- простые базовые блюда
- быстрые завтраки
- популярные рецепты

если это помогает:

- уменьшить сложность готовки
- использовать те же ингредиенты
- сделать список покупок проще.

---

## Правила составления плана закупок

План закупок должен строиться в два этапа:

1. Сначала агрегировать все ингредиенты на всю неделю.
2. Потом делить их на закупки по сроку хранения и разумности, а не просто по дням использования.

План закупок является ориентиром на момент генерации недели, а не строгим источником истины после всех последующих изменений.

Если по ходу недели `plan.yml` корректируется под фактическое питание, агент не обязан автоматически пересчитывать закупки или приводить их в полное соответствие с обновлённым фактом, если пользователь отдельно этого не просил.

### Категории закупки

#### 1. Купить сразу на неделю

Продукты, которые нормально хранятся 5–7 дней и дольше.

Примеры:

- яйца
- лук
- морковь
- картофель
- рис
- гречка
- паста
- овсянка
- гранола
- орехи
- арахисовая паста
- томатный соус

Такие продукты не нужно дробить на несколько закупок.

#### 2. Купить в первой половине недели и при необходимости докупить

Продукты со средним сроком годности.

Примеры:

- хлеб
- йогурт
- кефир
- молочные продукты
- помидоры
- огурцы
- кабачки
- брокколи
- ягоды
- зелень

#### 3. Купить ближе к дню приготовления

Очень скоропортящиеся продукты или продукты, которые лучше брать максимально свежими.

Примеры:

- свежее филе рыбы
- листья салата
- некоторые готовые молочные продукты с коротким сроком хранения

### Дополнительные правила

- Не дробить стабильные ингредиенты на несколько закупок.
- Яйца, крупы, пасту, лук, морковь и подобные продукты покупать сразу суммарно на неделю.
- Если ингредиент обычно продаётся упаковкой, допустимо округлять до упаковки.
- План закупок должен быть удобен для человека, а не только математически точен.

Если продукт обычно продаётся упаковкой, допустимо округлять количество до ближайшей упаковки.

Примеры:

- яйца: 15 → купить упаковку 20
- йогурт: 150 г → купить 1 упаковку
- кефир: 250 мл → купить бутылку
- авокадо: 140 г → купить 1–2 штуки

---

## Добавление новых рецептов

Чтобы добавить рецепт:

1. создать директорию `recipes/<meal_type>/<recipe_id>/`
2. создать файлы `recipe.yml` и `README.md`
3. соблюдать структуру `recipe.yml`
4. использовать реалистичные порции
5. избегать сложных рецептов
6. не использовать протеиновые порошки

---

## Что агент делать не должен

Агент не должен:

- придумывать рецепты вне репозитория
- использовать `_template` как реальные рецепты
- генерировать экстремально низкокалорийные планы
- строить меню из одного блюда на всю неделю
- вводить сложные системы учёта калорий или ингредиентов

---

## Критерий хорошего результата

Хороший meal plan — это план, который пользователь:

- реально соблюдает
- может готовить при рабочей нагрузке
- может легко закупить
- не бросает к середине недели
